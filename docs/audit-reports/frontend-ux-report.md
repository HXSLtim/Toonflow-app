# 前端体验审计报告

> ⚠️ 历史文档：本报告记录迁移前 UX 审计结果，包含 Electron 术语；当前架构已迁移为 `api/` + `web/` 纯 Web 模式。

**审计日期:** 2026-03-02
**审计人员:** Frontend UX Specialist

## 执行摘要

Toonflow 项目采用 Electron + Express 后端架构，前端为单页应用。审计发现 3 个 P0 级严重问题、5 个 P1 级高优先级问题和多个中低优先级问题。主要问题集中在启动流程端口不一致、错误处理不完善、用户反馈缺失等方面。

## 发现的问题

### P0 - 严重问题（阻塞用户使用）

1. **Electron 启动端口配置不一致导致应用无法正常工作**
   - **影响范围:** 所有 Electron 客户端用户
   - **复现步骤:**
     1. 启动 Electron 应用
     2. 后端服务在动态端口启动（如 60123）
     3. 前端窗口硬编码连接 60000 端口
     4. 应用无法连接后端服务
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\scripts\main.ts:37-38`
   - **问题代码:**
     ```typescript
     const port = await startServe(false);  // 返回实际端口
     createMainWindow(60000);               // 硬编码使用 60000
     ```
   - **修复建议:** 将实际端口传递给 `createMainWindow`
     ```typescript
     const port = await startServe(false);
     createMainWindow(port);  // 使用实际端口
     ```

2. **服务启动失败时用户无任何错误提示**
   - **影响范围:** 遇到端口占用或服务启动失败的用户
   - **复现步骤:**
     1. 占用 60000 端口
     2. 启动应用
     3. 应用窗口打开但无法连接，用户不知道发生了什么
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\scripts\main.ts:36-43`
   - **修复建议:**
     - 在服务启动失败时显示错误对话框
     - 提供重试或更改端口的选项
     - 记录详细错误日志供用户排查

3. **登录失败时返回 400 错误但消息不明确**
   - **影响范围:** 首次使用或登录失败的用户
   - **复现步骤:**
     1. 输入错误的用户名或密码
     2. 收到"登录失败"提示，无法判断是用户名错误还是密码错误
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\src\routes\other\login.ts:27`
   - **修复建议:** 区分用户不存在和密码错误的情况，提供更明确的提示

### P1 - 高优先级问题（严重影响体验）

1. **视频生成异步处理缺少进度反馈机制**
   - **影响范围:** 所有使用视频生成功能的用户
   - **复现步骤:**
     1. 提交视频生成请求
     2. 接口立即返回 200
     3. 用户不知道视频生成进度，只能不断刷新查询
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\src\routes\video\generateVideo.ts:119-122`
   - **修复建议:**
     - 实现 WebSocket 推送生成进度
     - 在数据库中记录详细的生成阶段（初始化、处理中、合成中等）
     - 前端显示进度条和预计剩余时间

2. **图片生成失败时状态更新但用户无法获知失败原因**
   - **影响范围:** 使用角色/场景/道具图片生成的用户
   - **复现步骤:**
     1. 提交图片生成请求
     2. AI 服务返回错误
     3. 数据库状态更新为"生成失败"，但前端可能不知道
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\src\routes\assets\generateAssets.ts:183-188`
   - **修复建议:**
     - 在数据库中添加 `errorReason` 字段记录失败原因
     - 前端轮询时显示失败原因
     - 提供重试按钮

3. **WebSocket 连接错误处理不完善**
   - **影响范围:** 使用分镜聊天和剧本生成的用户
   - **复现步骤:**
     1. 建立 WebSocket 连接
     2. 发送格式错误的数据
     3. 连接直接关闭，用户体验中断
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\src\routes\storyboard\chatStoryboard.ts:114-117`
   - **修复建议:**
     - 发送错误消息但不关闭连接
     - 允许用户重新发送正确的数据
     - 添加连接重连机制

4. **文件上传缺少大小和格式验证**
   - **影响范围:** 上传小说、图片的用户
   - **复现步骤:**
     1. 上传超大文件或错误格式文件
     2. 服务器处理失败或超时
     3. 用户等待很久才收到错误
   - **问题位置:** `C:\Users\a2778\Desktop\Code\Toonflow-app\src\app.ts:24-25`（100MB 限制过大）
   - **修复建议:**
     - 前端添加文件大小和格式预检查
     - 后端根据文件类型设置合理的大小限制
     - 显示上传进度条

5. **错误消息混用中英文且不一致**
   - **影响范围:** 所有用户
   - **复现步骤:** 查看各个接口的错误返回
   - **问题示例:**
     - `"Not Found"` (英文)
     - `"登录失败"` (中文)
     - `"视频配置不存在"` (中文)
   - **修复建议:**
     - 统一使用中文错误消息
     - 建立错误码体系（如 `ERR_LOGIN_FAILED`）
     - 支持国际化（i18n）

### P2 - 中优先级问题（影响体验但有变通方案）

1. **首次启动无引导流程**
   - 用户首次打开应用不知道如何开始
   - 建议添加欢迎页面和快速入门引导

2. **AI 模型配置界面缺少连接测试功能**
   - 用户配置 API Key 后不知道是否正确
   - 建议添加"测试连接"按钮

3. **长时间操作无加载状态指示**
   - 剧本生成、分镜生成等耗时操作缺少明确的加载提示
   - 建议添加全局加载遮罩和操作提示

4. **数据库操作缺少事务保护**
   - 批量插入小说章节时如果中途失败会导致数据不一致
   - 建议使用数据库事务确保原子性

5. **缺少操作确认对话框**
   - 删除项目、清空数据等危险操作无二次确认
   - 建议添加确认对话框防止误操作

6. **响应式布局支持不明确**
   - Electron 窗口固定大小 900x600，不支持调整
   - 建议支持窗口缩放和最小尺寸限制

### P3 - 低优先级问题（优化建议）

1. **日志输出混乱**
   - 控制台日志包含调试信息（如 `%c Line:30 🥓 url`）
   - 建议使用统一的日志系统，区分开发和生产环境

2. **缺少键盘快捷键支持**
   - 建议添加常用操作的快捷键（如 Ctrl+N 新建项目）

3. **缺少暗色模式支持**
   - 建议添加主题切换功能

4. **缺少离线提示**
   - 网络断开时用户不知道为什么操作失败
   - 建议添加网络状态检测和提示

5. **表单验证提示不友好**
   - Zod 验证错误消息格式化不够友好
   - 建议自定义错误消息模板

## 优秀实践

1. **使用 Zod 进行参数验证**
   - 统一的 `validateFields` 中间件确保 API 参数安全
   - 位置: `C:\Users\a2778\Desktop\Code\Toonflow-app\src\middleware\middleware.ts`

2. **WebSocket 实时通信**
   - 分镜聊天和剧本生成使用 WebSocket 提供流式体验
   - 事件驱动架构清晰，易于扩展

3. **异步任务处理**
   - 视频生成采用异步处理，避免阻塞用户操作
   - 立即返回任务 ID，用户可以继续其他操作

4. **统一的响应格式**
   - 使用 `success()` 和 `error()` 函数统一 API 响应格式
   - 位置: `C:\Users\a2778\Desktop\Code\Toonflow-app\src\lib\responseFormat.ts`

5. **JWT 认证机制**
   - 使用 JWT 保护 API，支持 Header 和 Query 两种传递方式
   - 白名单机制灵活

## 改进建议

### 短期（1-2 周）
1. 修复 P0 级问题，确保应用基本可用
2. 添加关键操作的进度反馈
3. 统一错误消息语言和格式

### 中期（1 个月）
1. 完善错误处理和用户反馈机制
2. 添加首次使用引导
3. 实现操作确认对话框
4. 优化 WebSocket 连接稳定性

### 长期（2-3 个月）
1. 建立完整的国际化支持
2. 实现主题切换功能
3. 添加键盘快捷键
4. 优化响应式布局
5. 建立用户行为分析和错误追踪系统
